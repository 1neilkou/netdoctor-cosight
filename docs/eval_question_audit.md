# Contest Solver 模拟题人工审计总结

本文档对 `contest_solver/data/sample_questions.json` 中 10 道模拟题进行人工审计。审计目标是识别参考答案、期望工具、期望轨迹点对自动评分的适配风险；不修改题库源文件。

## 重点审计结论

1. **Q003 存在自动评估冲突风险。** 参考答案的正确核心是“只触发规则A/MAJOR”。LLM 若给出简洁答案，未逐项说明 B/C 不触发，也应判为正确。另一个风险是 expected_tools 只写 `question_parser`，但真实系统路由并执行 `rule_evaluator` 是合理行为。

2. **Q006 的参考答案解释性内容偏多。** 自动语义评分若使用 token overlap，可能把“时域 14 个 OFDM 符号”“3GPP TS 38.101”“TCP重传率”“视频通话卡顿”等扩展解释当作必要要点，从而低估覆盖了关键事实的答案。建议人工评分时只把关键事实作为硬性项。

3. **Q008 不应因缺少 CELL_P/CELL_Q 字面量低分。** 题目的核心是阈值分级、CIO 根因排除、neighbor relation abnormal 的根因推断，以及后续排查方向。答案如果语义完整但未复述小区 ID，应保留较高分。

4. **Q010 的 expected_trace_points 对自动评分偏细。** 复杂规划题本身存在多种合理阶段划分和表达方式。若答案满足 4月28日许可申请、4月29-30日调整验证、邻区远程处理、DAS/天线现场处理、4小时观察、节中值守与风险预案，即使未逐字命中所有 trace point，也应人工判为合理。

## 建议

- 对 Q003/Q008/Q010 采用“关键结论覆盖”优先于严格文本匹配。
- 对 Q006 建议将评分关键点拆为：`12子载波`、`100MHz/30kHz 对应 273 PRB`、`PRB高利用率导致速率下降/时延增加/接入或体验劣化`。
- 对工具评估，允许实际系统调用 `rule_evaluator`、`task_planner` 等额外合理工具；expected_tools 更适合作为“最低覆盖要求”，不宜作为排他集合。
