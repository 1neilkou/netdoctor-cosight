# 网络知识库 — NetDoctor 参考手册

## 1. PRB 利用率过高

**定义**：PRB（Physical Resource Block）利用率是指已分配的物理资源块占总可用资源块的百分比，分为上行（UL）和下行（DL）两个维度。

**阈值参考**：
- 正常：< 70%
- 告警：70%~85%（建议关注）
- 危险：> 85%（需干预）
- 饱和：100%（业务严重受损）

**含义与影响**：
- 上行 PRB 高：用户上行速率被压缩，语音（AMR-WB）、视频会议、直播上传等实时业务卡顿；调度器无法为新用户分配资源，接入成功率下降。
- 下行 PRB 高：下行吞吐量受限，视频流播放卡顿，大文件下载慢；容量接近饱和时调度延迟增加。
- TDD 特殊性：上下行 PRB 利用率不对称时（如 UL 高、DL 正常），需评估帧结构调整可行性；盲目调整会影响整网同步，需全网协调。

**常见原因**：
1. 热点事件（演唱会、展览）导致话务突发
2. 写字楼上班高峰上行业务集中
3. 用户大量使用云备份、P2P 下载等高带宽应用
4. 站点故障导致话务溢出至邻区

## 2. RSRP / SINR 异常

### RSRP（参考信号接收功率）

**定义**：衡量服务小区信号强度，单位 dBm。

**阈值参考**：
| 等级 | RSRP 范围 | 用户体验 |
|------|-----------|---------|
| 优秀 | > -80 dBm | 满速率，业务流畅 |
| 良好 | -80 ~ -90 dBm | 基本正常，少量重传 |
| 一般 | -90 ~ -100 dBm | 速率下降，有感知 |
| 差 | -100 ~ -110 dBm | 严重劣化，频繁重传 |
| 极差 | < -110 dBm | 接近脱网，掉话风险高 |

**RSRP 异常含义**：RSRP 低于 -98 dBm 通常意味着覆盖不足（弱覆盖），可能由以下原因导致：天线故障/倾角过大、站间距过大（覆盖空洞）、建筑物遮挡、RRU 功率降额。

### SINR（信号与干扰加噪声比）

**定义**：衡量信号质量，反映信号在干扰和噪声背景下的可解调能力，单位 dB。

**阈值参考**：
| 等级 | SINR 范围 | 调制方式 |
|------|-----------|---------|
| 优秀 | > 20 dB | 256QAM |
| 良好 | 10~20 dB | 64QAM |
| 一般 | 3~10 dB | 16QAM / QPSK |
| 差 | 0~3 dB | QPSK（低阶） |
| 极差 | < 0 dB | 几乎无法解调 |

**SINR 异常含义**：SINR 低通常由同频干扰、邻频干扰、外部干扰（非授权频段设备）、越区覆盖叠加、或 RSRP 极低导致底噪抬升引起。

**RSRP 与 SINR 组合判断**：
- RSRP 低 + SINR 低 → 弱覆盖叠加干扰，需天线优化 + 频率规划
- RSRP 正常 + SINR 低 → 纯干扰问题，需排查干扰源
- RSRP 低 + SINR 正常 → 覆盖空洞但无竞争干扰，需补点或调整天线

## 3. 上行速率下降常见原因

**判断依据**：`ul_throughput_mbps` 持续低于基线水平（该小区历史均值），结合 `prb_usage_ul` 和 `sinr` 综合判断。

| 原因 | 特征指标组合 | 处理方向 |
|------|------------|---------|
| 上行 PRB 拥塞 | prb_usage_ul > 90，sinr 正常 | 话务均衡、帧结构调整、扩容 |
| 上行干扰 | sinr < 5，rsrp 正常，ul_throughput 低 | 干扰排查、频率规划 |
| 弱覆盖 | rsrp < -98，sinr < 3 | 天线调整、补点 |
| 传输链路劣化 | 误码告警，切换成功率也下降 | 光纤/传输设备排查 |
| 硬件故障 | RSRP/SINR 短时急速恶化，告警 hardware_warning | 现场排查 RRU/天馈 |
| 终端侧问题 | 仅个别用户投诉，整体 KPI 正常 | 终端层面排查 |

**注意**：上行速率下降往往滞后于 PRB 利用率上升，监控时应提前设置 PRB > 85% 的预警，而非等到吞吐量已明显下降才响应。

## 4. 切换成功率下降常见原因

**定义**：切换成功率 = 成功完成切换次数 / 切换尝试次数 × 100%。正常水平应 > 97%。

**下降原因及特征**：

| 原因类别 | 典型特征 | 处置方法 |
|---------|---------|---------|
| 邻区关系缺失/错误 | 切换失败集中于特定方向，SON ANR 日志有缺失邻区提示 | 补充邻区关系，开启 SON ANR |
| 切换参数不当（A3 CIO 过大） | 乒乓切换比率高，或切换过晚导致 RLF | 调整 A3 偏移量，建议 0~+3 dB |
| 目标小区过载 | 切换失败伴随目标小区 PRB 高 | 负载均衡，限制切入 |
| 传输链路问题 | 伴随传输误码告警，切换信令丢失 | 传输设备排查 |
| 邻区 PCI 变更 | 特定邻区切换失败率突增，邻区信息不匹配 | 更新邻区 PCI 配置 |
| 新开通小区配置错误 | 新站开通后立即出现，RSRP/SINR 正常但切换失败 | 检查 QoS 映射、TA offset、网络切片配置 |
| 硬件/覆盖恶化 | RSRP/SINR 同步劣化，切换成功率随之下降 | 覆盖优化或硬件排查 |

**关键阈值**：
- 切换成功率 < 95%：需关注并分析原因
- 切换成功率 < 90%：需立即干预
- 切换成功率 < 85%：严重，影响用户体验，优先级最高

## 5. 告警与 KPI 的关联规则

以下规则用于辅助 NetDoctor 智能诊断，将告警类型与 KPI 指标联合分析，提升根因定位准确率。

### 规则 R1：上行拥塞告警
```
IF alarm_type = 'cell_overload'
   AND prb_usage_ul > 90
   AND ul_throughput_mbps < 3
THEN root_cause_candidate = '上行容量饱和'
     action = '话务均衡 / 帧结构评估 / 扩容'
```

### 规则 R2：干扰导致弱覆盖
```
IF alarm_type = 'high_interference'
   AND rsrp < -98
   AND sinr < 3
THEN root_cause_candidate = '弱覆盖叠加同频干扰'
     action = '天线优化 + 干扰源排查 + 频率规划'
```

### 规则 R3：切换失败 — 邻区关系异常
```
IF alarm_type = 'neighbor_relation_abnormal'
   AND handover_success_rate < 90
   AND drop_rate > 2.0
THEN root_cause_candidate = '邻区关系缺失或切换参数不当'
     action = '核查邻区关系表，调整 A3 CIO 参数'
```

### 规则 R4：传输劣化引发切换失败
```
IF alarm_type = 'transmission_degradation'
   AND handover_success_rate < 92
   AND sinr 呈下降趋势（连续 3 个采样点）
THEN root_cause_candidate = '传输链路误码或中断'
     action = '传输设备及光纤排查'
```

### 规则 R5：硬件故障引发覆盖骤降
```
IF alarm_type = 'hardware_warning'
   AND rsrp 短时下降 > 10 dBm
   AND sinr < 0
THEN root_cause_candidate = 'RRU / 天馈硬件故障'
     action = '立即派单现场排查 RRU 及天馈系统'
     priority = 'P1（最高优先级）'
```

### 规则 R6：无告警但 KPI 劣化（潜在隐患）
```
IF alarm_type = None
   AND prb_usage_ul > 85
   AND ul_throughput_mbps < 5
   AND user_complaints > 8
THEN root_cause_candidate = '上行容量逼近瓶颈（未达告警阈值）'
     action = '提前预警，评估扩容或参数调优'
```

## 6. 参数优化风险提示

在执行任何网络参数调整前，必须充分评估以下风险：

### 6.1 天线参数调整

| 参数 | 调整方向 | 潜在风险 |
|------|---------|---------|
| 下倾角增大（如 4°→8°） | 减少越区覆盖 | 本小区覆盖半径收缩，边缘用户信号变弱，可能产生新的覆盖空洞 |
| 下倾角减小 | 扩大覆盖范围 | 与远端小区产生同频干扰，SINR 劣化 |
| 发射功率增加 | 改善覆盖 | 对邻区干扰增加，整体 SINR 环境恶化；需配套频率规划 |
| 方位角调整 | 改变覆盖方向 | 可能导致原覆盖区域产生弱覆盖，切换关系需同步更新 |

**建议**：天线参数调整应在仿真工具中预演后实施，调整范围单次不超过 3°，且需同步观察邻区指标变化。

### 6.2 切换参数调整

| 参数 | 调整方向 | 潜在风险 |
|------|---------|---------|
| A3 CIO 减小 | 提前触发切换 | 切换过于频繁，乒乓切换比率上升，信令开销增加 |
| A3 CIO 增大 | 延迟切换触发 | 用户在服务小区边缘停留过久，SINR 持续劣化后切换失败 |
| TTT（触发时间）缩短 | 加快切换响应 | 对信号波动敏感，短时干扰可能触发不必要切换 |
| TTT 延长 | 减少误触发 | 切换响应迟缓，快速移动用户体验差 |

**建议**：切换参数调整必须在非业务高峰期实施，并设置自动回滚机制（调整后 2 小时内未改善则回退）。

### 6.3 TDD 帧结构调整

- **风险等级：极高**，必须全网协调统一修改，不允许单站调整
- 调整上下行时隙比会改变小区的上下行容量比，影响全网所有用户
- 需提前通知运维、网络规划、无线优化等多部门联合评审
- 必须在深夜低话务时段实施，并准备快速回滚预案

### 6.4 频率/PCI 规划调整

- PCI 修改会导致所有相关邻区的邻区关系失效，必须同步更新邻区配置
- 跨运营商邻区 PCI 变更需提前通知对端，避免单方面变更引发切换失败
- 频率调整需重新进行干扰分析，评估对周边小区 SINR 的影响

### 6.5 通用操作规范

1. **变更前**：记录当前基准 KPI（RSRP、SINR、切换成功率、掉话率、吞吐量），作为效果对比依据
2. **变更中**：实时监控告警和关键 KPI，出现劣化立即暂停
3. **变更后**：观察时间不少于 24 小时（覆盖工作日高峰期），确认 KPI 稳定后方可关闭工单
4. **回滚条件**：任一关键指标（切换成功率 < 90%，掉话率 > 3%，SINR 下降 > 5 dB）触发自动或手动回滚
